1. Executive Summary

The EBO-SACCO Credit Scoring System is a supervised machine learning framework that predicts whether a loan applicant will repay (class 1) or default (class 0). It was trained on 17,000 historical loan records (2021–2025) from EBO-SACCO's portfolio in Western Uganda, as part of an MSc IT research project at Victoria University, Kampala.

Four algorithms were trained and benchmarked — Logistic Regression, Decision Tree, Random Forest, and XGBoost. Predictions are served through a Flask REST API with:

  • SHAP explainability — every prediction returns its top decision drivers in plain language.
  • Business rule guardrails — hard policy limits (age, DTI, LTV, savings multiplier) that override the model and can never be out-voted by it.
  • Input validation — bounded, typed request checking before any inference runs.

3. Dataset

SourceEBO-SACCO loan history (Excel)
Records17,000 loans (12,952 train / 4,048 test)
Period2021–2025
TargetLoanStatus — 0 = Default, 1 = Repaid
Raw fieldsDemographics, financials, loan terms, repayment behaviour, dates

Columns removed before training

  • Identifiers: BorrowerID, Name (privacy; no predictive value)
  • Zero variance: MobileMoneyTransactions
  • Date columns: consumed into temporal features, then dropped
Post-disbursement features. Six behavioural features (missed installments, days in arrears, repayment counts and values, deposits, account activity) describe activity after a loan is disbursed. They are deliberately included to support the research's portfolio-monitoring use case and dominate the model's learned signal. For a strictly pre-approval deployment they should be excluded (a commented-out drop list exists in config.py). At prediction time the API imputes them with training-set medians, since a new applicant has no repayment history.

4. Preprocessing & Feature Engineering

The pipeline produces 43 model features from the raw data.

Temporal features

ApprovalYear, ApprovalMonth, ApprovalQuarter (from the approval date) and DisbursementDelay (days between approval and first disbursement).

Engineered financial ratios

FeatureFormula
LoanToIncomeRatioLoanAmount / AverageMonthlyIncome
LoanToCollateralRatioLoanAmount / TotalCollateralValue
SavingsToLoanRatioCurrentSavingsBalance / LoanAmount
CollateralCoverageTotalCollateralValue / LoanAmount
DTI (%)(LoanAmount / LoanDuration) / AverageMonthlyIncome × 100
LTV (%)LoanAmount / TotalCollateralValue × 100

All ratios guard against division by zero (default to 0).

Categorical encoding

  • Gender, PreviousDefaultHistory, EmploymentStatus — binary encoded
  • LoanType — grouped to top 10 categories + “Other”, then one-hot encoded
  • MaritalStatus — mapped to MARRIED / SINGLE / OTHER, then one-hot encoded
  • RepaymentSchedule — one-hot encoded (base category dropped)

Cleaning

  • Missing values: numeric → median; categorical → mode
  • Outlier capping at the 99th percentile for income, savings, loan amount, collateral, repayment value
  • Scaling with StandardScaler fitted on training data only

5. Training Methodology

Temporal train/test split

To simulate real deployment (predicting future loans from past data), the split is temporal rather than random: train on 2021–2024 (12,952 loans), test on 2025 (4,048 loans).

Class imbalance handling

  • SMOTE oversampling — Logistic Regression, Decision Tree, Random Forest
  • scale_pos_weight — XGBoost, trained on original un-resampled data

Cross-validation & hyperparameters

5-fold stratified cross-validation scored by ROC-AUC. Fixed random seed (42) throughout for reproducibility.

ModelKey settings
Logistic RegressionC=1.0, lbfgs, max_iter=1000
Decision Treemax_depth=10, min_samples_split=20, min_samples_leaf=10, balanced class weights
Random Forest200 trees, max_depth=12, min_samples_leaf=3, max_features=sqrt, balanced class weights
XGBoost250 trees, max_depth=5, learning_rate=0.1, subsample=0.8, colsample_bytree=0.8, scale_pos_weight

Interpreting the champion (XGBoost)

  • 93.2% default recall — catches 93 of every 100 loans that would default. Missing a defaulter is the costly error for a lender, so this is the headline number.
  • 95.2% precision — when it approves, it is almost always right.
  • 70.8% repaid recall — the trade-off: roughly 29% of creditworthy applicants are flagged as risky. A deliberately conservative operating point for a capital-constrained microfinance institution.
  • The gap between CV AUC (0.970) and test AUC (0.879) reflects temporal drift — 2025 loans differ in distribution from 2021–2024. This is expected, and is precisely why a temporal split gives an honest estimate.

McNemar's test (with continuity correction) confirms the champion's edge over each other model is statistically significant rather than noise.

The top six drivers are post-disbursement behavioural features (see §3). Among application-time features, the strongest signals are loan duration, income, loan amount, and collateral value. Per prediction, the API returns the top 5 drivers (excluding features unknown at application time) with a direction — e.g. “LoanDuration — increases risk” — which this webapp renders in the result panel.

Decision logic: guardrail violations always deny (DENIED — POLICY RULE); otherwise approve if P(repaid) ≥ 0.5, else LOAN DENIED — HIGH RISK. confidence reports P(repaid) on approvals and P(default) on denials. Errors return 400 with a message field.

Feature construction at inference. The API rebuilds the 43-feature vector from the 12 input fields: ratios recomputed with the training formulas, approval date set to today, Gender fixed to 0 (not collected, for fair-lending compliance), one-hot groups defaulted to training base categories, post-disbursement features imputed with training-set medians, and everything scaled with the persisted training scaler.

9. Business Rule Guardrails

Hard policy limits applied after the model. Any violation results in denial regardless of the model's score, and violated rules are itemized in the response.

RuleThreshold
Applicant age18–75 years
Debt-to-income ratio (DTI)≤ 60%
Loan-to-value ratio (LTV)≤ 100%
Loan-to-savings capLoan ≤ 3× savings balance

Local

python train.py          # retrain (requires dataset in project root)
python eval_model.py     # optional: McNemar + SHAP evaluation
python api.py            # serve API + webapp at http://localhost:5000

Production (Render.com)

  • gunicorn with a single worker so the 4 models and SHAP explainers load into memory exactly once
  • Health check on /health; PORT and public URL injected by the platform; CORS extends automatically
  • The Flask API serves this webapp at the root path

Logging

logs/training.log traces each training run; logs/api.log records every prediction with model, decision, P(default), DTI, LTV, and violation count — an audit trail.

11. Limitations & Ethical Considerations

  1. Post-disbursement signal dominance. The strongest predictors are repayment-behaviour features unavailable for genuinely new applicants; at inference these are median-imputed, so first-time applicants are effectively scored on application-time features only. For a pure pre-approval product, retrain with those features excluded and re-benchmark.
  2. Temporal drift. Test AUC (0.879) sits below CV AUC (0.970) because loan behaviour shifts year to year. Retrain at least annually and monitor for drift.
  3. Fairness. Gender exists in training data but is neutralized at inference. A formal disparate-impact analysis across gender, age, and marital-status groups is recommended before regulatory-grade deployment.
  4. Threshold choice. The 0.5 threshold favours default detection over approval volume; tune it to the institution's actual cost of default vs. rejected good customers.
  5. Scope. Trained exclusively on EBO-SACCO's Western-Uganda portfolio; performance does not transfer elsewhere without revalidation.
  6. Decision support, not automation. Outputs are recommendations with explanations; final decisions should remain with credit officers.

12. Retraining Procedure

  1. Place the updated dataset in the project root (or pass --data path.xlsx)
  2. Run python train.py — regenerates all models, plots, and metadata
  3. Review models/model_comparison.csv and plots/ for regressions
  4. Run python eval_model.py — confirm significance (McNemar) and inspect SHAP shifts
  5. Update the UNKNOWN_AT_APPLICATION medians in api.py to match the new training data
  6. Bump MODEL_VERSION in config.py; restart the API to load new artifacts
